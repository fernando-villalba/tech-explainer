# Kubebuilder: The Scaffolding Machine Behind Kubernetes Operators

Every Kubernetes operator tutorial starts the same way. Run `kubebuilder init`. Run `kubebuilder create api`. Fill in the Reconcile function. Deploy. It feels like Rails or Spring Boot -- a framework that you build on top of, that stays under your code at runtime.

That mental model is wrong. And it leads to confusion about where kubebuilder ends, where controller-runtime begins, what controller-gen does, and why `make manifests` matters.

Kubebuilder is not a framework. It is a scaffolding tool. It generates a project, writes every file you need to get started, and then has zero runtime presence. There is no `kubebuilder` import in your Go code. There is no `kubebuilder` binary running in your cluster. The moment scaffolding finishes, kubebuilder's job is done. The actual framework your operator runs on is controller-runtime. The actual code generator that reads your markers and produces CRD manifests is controller-gen. Kubebuilder just writes the starter code that wires them together.

Three tools, one operator. Understanding which tool does what is the key to understanding the whole system. Everything that follows is organized around these three tools.

If you're wondering "what about Operator SDK?" -- Operator SDK is kubebuilder. It imports kubebuilder's CLI as a library, registers the same Go scaffolding plugin, and adds extra commands for OLM distribution and extra plugins for Ansible and Helm operators. For Go operators, `operator-sdk init` and `kubebuilder init` produce identical output. The [Operator SDK explainer](01-operator-sdk.md) covers what it adds and when you'd use it instead.

## Tool 1: [Kubebuilder](https://github.com/kubernetes-sigs/kubebuilder) -- The Scaffolder

Kubebuilder is a CLI that runs on your laptop. You invoke it a handful of times when starting a project or adding new APIs. It writes files and then gets out of the way. It has three commands that matter.

### `kubebuilder init`

Run `kubebuilder init --domain example.com --repo github.com/you/your-operator` in an empty directory and it scaffolds the entire project skeleton:

```
cmd/main.go              -- the entry point (wires up the controller-runtime Manager)
api/                     -- empty, waiting for kubebuilder create api
internal/controller/     -- empty, waiting for kubebuilder create api
config/
  default/               -- kustomize overlay that combines everything
  manager/               -- Deployment manifest for the operator pod
  rbac/                  -- ServiceAccount, ClusterRoles, bindings
  crd/                   -- empty, will hold CRD YAML generated by controller-gen
  webhook/               -- empty, will hold webhook config generated by controller-gen
  certmanager/           -- cert-manager integration for webhook TLS
  network-policy/        -- restrict operator pod network access
Makefile                 -- build system that calls controller-gen and kubectl
Dockerfile               -- multi-stage Go build
PROJECT                  -- kubebuilder's own metadata file
go.mod                   -- controller-runtime as the primary dependency
```

The `PROJECT` file is kubebuilder's memory. It tracks which APIs you've scaffolded, which webhooks exist, which plugins are active. Every subsequent `kubebuilder` command reads this file to know what already exists and where to inject new code.

The `config/` directory is a tree of Kustomize bases and overlays. `config/default/kustomization.yaml` is the root -- it pulls in manager, RBAC, CRD, webhook, and cert-manager configs and composes them into a single deployable manifest. By default, the webhook and cert-manager sections are commented out. You uncomment them when you add webhooks. This is a manual step that trips up many developers -- they scaffold webhooks but forget to enable them in the Kustomize overlay, then wonder why admission never fires.

### `kubebuilder create api`

This is the command that does the most. Run `kubebuilder create api --group crew --version v1 --kind Captain` and it writes three things.

**First, the types.** A file at `api/v1/captain_types.go` defines what a Captain custom resource looks like:

```go
// CaptainSpec defines the desired state of Captain
type CaptainSpec struct {
    // foo is an example field of Captain. Edit captain_types.go to remove/update
    // +optional
    Foo *string `json:"foo,omitempty"`
}

// CaptainStatus defines the observed state of Captain.
type CaptainStatus struct {
    // +listType=map
    // +listMapKey=type
    // +optional
    Conditions []metav1.Condition `json:"conditions,omitempty"`
}

// +kubebuilder:object:root=true
// +kubebuilder:subresource:status

// Captain is the Schema for the captains API
type Captain struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitzero"`
    Spec   CaptainSpec   `json:"spec"`
    Status CaptainStatus `json:"status,omitzero"`
}
```

Every CRD in Kubernetes follows this shape. `Spec` is what the user wants. `Status` is what the controller reports. The scaffolded types are deliberately minimal -- a `Foo` string placeholder and a `Conditions` slice. This is where you define your domain model. In the multigres-operator, the `MultigresClusterSpec` that started from this template grew to include connection strings, storage classes, backup schedules, PVC retention policies, cell topology, and template references.

Notice the `// +kubebuilder:object:root=true` and `// +kubebuilder:subresource:status` comments. These look like Go comments but they're markers -- structured metadata for controller-gen. Kubebuilder writes the initial markers. Controller-gen reads them later. We'll cover exactly what happens to them in Tool 2.

**Second, the controller.** A file at `internal/controller/captain_controller.go`:

```go
type CaptainReconciler struct {
    client.Client
    Scheme *runtime.Scheme
}

// +kubebuilder:rbac:groups=crew.example.com,resources=captains,verbs=get;list;watch;create;update;patch;delete
// +kubebuilder:rbac:groups=crew.example.com,resources=captains/status,verbs=get;update;patch
// +kubebuilder:rbac:groups=crew.example.com,resources=captains/finalizers,verbs=update

func (r *CaptainReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    _ = logf.FromContext(ctx)

    // TODO(user): your logic here

    return ctrl.Result{}, nil
}

func (r *CaptainReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&crewv1.Captain{}).
        Named("captain").
        Complete(r)
}
```

That empty `Reconcile` function is the blank canvas. The `// +kubebuilder:rbac:` comments above it are markers too -- they'll tell controller-gen what RBAC rules to generate. And `SetupWithManager` wires the controller into the controller-runtime Manager's event pipeline. We'll cover what both of these do when we get to their respective tools.

**Third, it modifies `cmd/main.go`.** Kubebuilder injects code at marked insertion points -- comments like `// +kubebuilder:scaffold:imports`, `// +kubebuilder:scaffold:scheme`, and `// +kubebuilder:scaffold:builder`. It adds an import for the new API package, registers the types with the runtime scheme, and wires up the controller:

```go
if err := (&controller.CaptainReconciler{
    Client: mgr.GetClient(),
    Scheme: mgr.GetScheme(),
}).SetupWithManager(mgr); err != nil {
    setupLog.Error(err, "Failed to create controller", "controller", "Captain")
    os.Exit(1)
}
```

Each `create api` call adds to the existing `main.go` rather than regenerating it. If you accidentally delete a scaffold marker comment, the next `create api` command fails because it can't find where to inject code.

### `kubebuilder create webhook`

Run `kubebuilder create webhook --group crew --version v1 --kind Captain --defaulting --validation` and it scaffolds a webhook file at `internal/webhook/v1/captain_webhook.go` with stub implementations:

```go
type CaptainCustomDefaulter struct{}

func (d *CaptainCustomDefaulter) Default(ctx context.Context, obj *crewv1.Captain) error {
    // TODO(user): fill in your defaulting logic.
    return nil
}

type CaptainCustomValidator struct{}

func (v *CaptainCustomValidator) ValidateCreate(ctx context.Context, obj *crewv1.Captain) (admission.Warnings, error) {
    // TODO(user): fill in your validation logic upon object creation.
    return nil, nil
}

func (v *CaptainCustomValidator) ValidateUpdate(ctx context.Context, oldObj, newObj *crewv1.Captain) (admission.Warnings, error) {
    // TODO(user): fill in your validation logic upon object update.
    return nil, nil

func (v *CaptainCustomValidator) ValidateDelete(ctx context.Context, obj *crewv1.Captain) (admission.Warnings, error) {
    // TODO(user): fill in your validation logic upon object deletion.
    return nil, nil

```

It also adds webhook marker comments (for controller-gen) and wires the webhook into `cmd/main.go` (for controller-runtime). Both of those will make sense after the next two sections.

### The plugin system

Kubebuilder is extensible through plugins. The `--plugins` flag on `kubebuilder init` determines which plugin handles scaffolding. The default is `go/v4`, which scaffolds a Go project targeting controller-runtime.

Each plugin implements interfaces for each command: `InitSubcommand`, `CreateAPISubcommand`, `CreateWebhookSubcommand`, `EditSubcommand`. The `go/v4` plugin produces everything described above. Other plugins shipped with the kubebuilder binary modify or extend the output:

- **kustomize/v2** -- generates the Kustomize `config/` directory structure
- **deploy-image/v1-alpha** -- scaffolds a controller that manages Deployments (not just CRDs)
- **grafana/v1-alpha** -- generates Grafana dashboards for operator metrics
- **helm/v2-alpha** -- generates a Helm chart from the Kustomize output (not the same as using Helm *as* the reconciler)
- **autoupdate/v1-alpha** -- scaffolds GitHub Actions for automated dependency updates

The plugin system's real power is that other projects can import kubebuilder as a library and register their own plugins. Operator SDK does exactly this -- it wraps kubebuilder's CLI and adds plugins for Ansible and Helm-based operators that are not available in the kubebuilder binary. The [Operator SDK explainer](01-operator-sdk.md) covers this in detail.

Plugins are tracked in the `PROJECT` file. Kubebuilder reads them to know which interfaces to call during each command.

### Where kubebuilder stops

After scaffolding, kubebuilder is done. It wrote Go files, a Makefile, Kustomize configs, and a Dockerfile. Every one of those files is yours to modify. There is no "regeneration" step that overwrites your changes. If you never run `kubebuilder` again, your project works fine.

The design doc makes this philosophy explicit:

> Prefer using go interfaces and libraries over-relying on code generation. Prefer using code generation over 1 time init of stubs. Prefer 1 time init of stubs over forked and modified boilerplate. Never fork and modify boilerplate.

Kubebuilder does the "1 time init of stubs." The other two tools handle the rest.

## Tool 2: [Controller-gen](https://github.com/kubernetes-sigs/controller-tools) -- The Code Generator

`controller-gen` is a CLI from the [controller-tools](https://github.com/kubernetes-sigs/controller-tools) project. It reads your Go source files, finds marker comments, and generates artifacts. It runs at build time through the Makefile, not at deploy time. It has no runtime presence.

It has six generators:

1. **`crd`** -- generates CRD YAML from your Go types and their validation markers
2. **`rbac`** -- generates ClusterRole YAML from `+kubebuilder:rbac:` markers
3. **`object`** -- generates `DeepCopy` method implementations for your types
4. **`webhook`** -- generates `MutatingWebhookConfiguration` and `ValidatingWebhookConfiguration` YAML from webhook markers
5. **`applyconfiguration`** -- generates typed builders for Server-Side Apply clients
6. **`schemapatch`** -- merges generated OpenAPI schemas into existing CRD manifests

Kubebuilder's Makefile wires up the first four. The other two are available if you need them -- you'd invoke them by adding arguments to the `controller-gen` command in your Makefile.

### What markers are

Markers are Go comments with a specific prefix that controller-gen parses as structured metadata. The Go compiler ignores them completely. They look like this:

```go
// +kubebuilder:validation:Minimum=0
// +kubebuilder:rbac:groups=apps,resources=deployments,verbs=get;list;watch
// +kubebuilder:object:root=true
// +kubebuilder:webhook:path=/mutate-crew-example-com-v1-captain,mutating=true,...
```

The naming is the single most confusing thing in this ecosystem. Every marker starts with `// +kubebuilder:` but kubebuilder does not read them. controller-gen reads them. The prefix is a historical artifact -- markers and controller-gen were developed as part of the kubebuilder project before controller-tools was split into its own repository. The name stuck. So when you write `// +kubebuilder:validation:Minimum=0`, you're talking to controller-gen, not kubebuilder. When you write `// +kubebuilder:rbac:groups=apps`, you're talking to controller-gen. Kubebuilder scaffolds the initial markers when you run `create api` or `create webhook`. From that point on, they belong to controller-gen. kubebuilder never looks at them again.

### `make generate` -- DeepCopy methods

Every Kubernetes object needs a `DeepCopyObject()` method. Writing these by hand would be tedious and error-prone. The Makefile's `generate` target runs:

```
controller-gen object:headerFile="hack/boilerplate.go.txt" paths="./..."
```

Controller-gen scans your types, finds structs with `// +kubebuilder:object:root=true`, and generates `zz_generated.deepcopy.go` files alongside your types. These files are always safe to overwrite -- they contain only generated code.

If you add a new type and forget `make generate`, your code compiles until something tries to use it through a `runtime.Object` interface. The error is cryptic: "does not implement runtime.Object." The fix is always `make generate`.

### `make manifests` -- CRDs, RBAC, and webhook configs

This is where most of the marker magic happens. The Makefile runs:

```
controller-gen rbac:roleName=manager-role crd webhook paths="./..." \
    output:crd:artifacts:config=config/crd/bases
```

Controller-gen reads every `.go` file in your project as text, finds lines starting with `// +kubebuilder:`, and emits three categories of YAML.

**CRD manifests from type markers.** Your Go struct fields, their types, their JSON tags, and their validation markers become an OpenAPI v3 schema inside the CRD YAML. Controller-gen writes these to `config/crd/bases/`.

```go
// +kubebuilder:default=1
// +kubebuilder:validation:Minimum=0
// +kubebuilder:validation:Maximum=100
// +optional
Size *int32 `json:"size,omitempty"`
```

This becomes a CRD field with `minimum: 0`, `maximum: 100`, `default: 1`. The API server enforces these rules before your controller ever sees the object. A user creating a Captain with `size: -1` gets rejected at the API level. Your Reconcile function never needs to check.

Structural markers on the type itself control higher-level CRD behavior:

```go
// +kubebuilder:object:root=true           -- this type is a CRD root kind
// +kubebuilder:subresource:status         -- enable the /status subresource
// +kubebuilder:printcolumn:name="Phase",type="string",JSONPath=".status.phase"
// +kubebuilder:resource:shortName=mgc     -- kubectl get mgc works
```

`+kubebuilder:subresource:status` is particularly important. Without it, any update to the object overwrites both spec and status. With it, spec and status have separate API endpoints. Your controller patches status without touching spec. Users update spec without clobbering status. The multigres-operator uses this on every CRD.

**RBAC manifests from RBAC markers.** Controller-gen aggregates every `+kubebuilder:rbac:` marker across every Go file and produces one `ClusterRole` at `config/rbac/role.yaml`.

```go
// +kubebuilder:rbac:groups=apps,resources=deployments,verbs=get;list;watch;create;update;patch;delete
// +kubebuilder:rbac:groups=core,resources=pods,verbs=get;list;watch
```

Kubebuilder scaffolds the initial RBAC markers for your primary CRD type. As you add `r.Get()`, `r.Create()`, `r.Patch()` calls for Deployments, Services, and ConfigMaps in your controller, you add corresponding RBAC markers. If you forget, your operator crashes at runtime with "forbidden" errors from the API server -- the ClusterRole doesn't grant the permissions your code needs.

**Webhook manifests from webhook markers.** The webhook marker that kubebuilder scaffolded:

```go
// +kubebuilder:webhook:path=/mutate-crew-example-com-v1-captain,mutating=true,
//   failurePolicy=fail,sideEffects=None,groups=crew.example.com,
//   resources=captains,verbs=create;update,versions=v1,
//   name=mcaptain.kb.io,admissionReviewVersions=v1
```

Controller-gen reads this and generates `MutatingWebhookConfiguration` or `ValidatingWebhookConfiguration` YAML in `config/webhook/manifests.yaml`. The path is deterministic: `/mutate-` or `/validate-` followed by the GVK with dots replaced by dashes.

### The critical detail

Markers are consumed at build time, not runtime. Controller-gen reads your `.go` files as text and emits YAML. The Go compiler ignores markers entirely. If you mistype a marker, your code compiles fine but `make manifests` produces wrong output or silently drops the marker. Run `make manifests` after every marker change and inspect the diff.

The only files controller-gen regenerates are:

- `zz_generated.deepcopy.go` -- from `make generate`
- `config/crd/bases/*.yaml` -- from `make manifests`
- `config/rbac/role.yaml` -- from `make manifests`
- `config/webhook/manifests.yaml` -- from `make manifests`

Everything else in the project is untouched. Your types, your controllers, your webhooks, your main.go -- controller-gen reads them but never writes them.

## Tool 3: [Controller-runtime](https://github.com/kubernetes-sigs/controller-runtime) -- The Runtime Framework

Controller-runtime is a Go library (`sigs.k8s.io/controller-runtime`). It's the only one of the three tools that runs in your cluster. Your `go.mod` imports it. Your binary links against it. It provides the Manager, the Reconciler interface, the cache, the client, the webhook server, leader election, and health probes.

The [controller-runtime explainer](../controller-runtime/00-overview.md) covers the internals in depth. Here, we'll focus on how the code kubebuilder scaffolded maps to controller-runtime's abstractions.

### The Manager

The `cmd/main.go` that `kubebuilder init` generated creates a Manager:

```go
mgr, err := ctrl.NewManager(ctrl.GetConfigOrDie(), ctrl.Options{
    Scheme:                 scheme,
    Metrics:                metricsserver.Options{...},
    WebhookServer:          webhook.NewServer(webhook.Options{...}),
    HealthProbeBindAddress: probeAddr,
    LeaderElection:         enableLeaderElection,
    LeaderElectionID:       "abc123.example.com",
})
```

The Manager is the top-level coordinator. It owns the cache (a local mirror of cluster state backed by informers), the client (reads from cache, writes to API server), the webhook server, health probes, metrics, and every controller. When the binary calls `mgr.Start(ctx)`, it boots the entire system in a precise sequence: HTTP servers first, then the cache syncs, then leader election runs, then controllers start processing. The [manager lifecycle deep-dive](../controller-runtime/03-manager-lifecycle.md) covers why this ordering matters.

### The Reconciler

The scaffolded controller embeds `client.Client` and implements the `Reconciler` interface:

```go
type CaptainReconciler struct {
    client.Client
    Scheme *runtime.Scheme
}

func (r *CaptainReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    // TODO(user): your logic here
    return ctrl.Result{}, nil
}
```

The `Request` contains only a namespace and name. No event type. No old object. No diff. This is the core design decision in controller-runtime: your Reconcile function reads current state, compares it to desired state, and closes the gap. Like a thermostat that reads the current temperature and adjusts, regardless of how many times the temperature changed or which direction it went.

In a real operator, this blank canvas fills in with the canonical pattern. The multigres-operator's `MultigresCluster` reconciler fetches the cluster object, checks if it still exists, reads its spec, computes what child resources should exist (Cells, TableGroups, TopoServer, global services), compares that to what does exist, and creates or updates the difference. Same function whether it was triggered by a user edit, a child object change, or a periodic resync.

### The Builder

The scaffolded `SetupWithManager` uses controller-runtime's builder pattern to wire the event pipeline:

```go
func (r *CaptainReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&crewv1.Captain{}).
        Named("captain").
        Complete(r)
}
```

`For(&Captain{})` means "watch Captain resources and enqueue their names when they change." Kubebuilder scaffolds this minimal setup. As your operator grows, you extend it:

- `Owns(&appsv1.Deployment{})` -- "when a Deployment I own changes, enqueue the owning Captain." Walks OwnerReferences automatically.
- `Watches(&corev1.ConfigMap{}, handler.EnqueueRequestsFromMapFunc(mapper))` -- "when a ConfigMap changes, run my custom mapper to decide which Captains to re-reconcile."
- `WithPredicates(predicate.GenerationChangedPredicate{})` -- "ignore status-only updates."

The multigres-operator's MultigresCluster controller watches its primary type, owns five child types, and watches three template types with custom mappers. The Shard controller owns seven child types. Every `Owns()` and `Watches()` call is pure controller-runtime API -- kubebuilder just generated the initial `For()` to get you started.

### Webhooks at runtime

The defaulter and validator stubs that kubebuilder scaffolded implement controller-runtime interfaces. At runtime, controller-runtime handles all the admission plumbing:

For the **defaulter**: controller-runtime decodes the admission request, deep-copies the original object, calls your `Default` function, computes the JSON patch between original and mutated, and sends the response. You never think about patches. You just modify the Go struct and the framework diffs it.

For the **validator**: controller-runtime decodes the request, calls `ValidateCreate`, `ValidateUpdate`, or `ValidateDelete` depending on the operation, and returns allow/deny based on whether you returned an error. In multigres, the validator enforces that shard counts are positive, storage class names exist, and template references resolve -- catching user errors before they reach the controller.

Webhooks are where you enforce invariants that CRD validation markers (controller-gen) can't express. Markers handle structural constraints: field ranges, required fields, regex patterns. Webhooks handle semantic constraints: cross-field validation, external lookups, business logic. Markers run in the API server. Webhooks run in your operator pod via controller-runtime's webhook server.

### What controller-runtime provides that kubebuilder doesn't

Everything that happens at runtime comes from controller-runtime. The cache that makes reads local memory lookups instead of API server calls. The workqueue that de-duplicates events so you don't reconcile 50 times when 50 events fire. The rate limiter with exponential backoff for failing objects. The leader election that ensures only one replica processes at a time. The split client that reads from cache but writes to the API server. The panic recovery that catches a crashing Reconcile function and requeues the item instead of killing the process.

Kubebuilder generated the code that calls into these systems. But if you deleted kubebuilder from your laptop and never ran it again, your operator would work identically. The runtime is controller-runtime.

## How the Three Tools Connect

The build pipeline makes the relationships concrete:

```
You edit Go types and add markers
         |
         v
    make generate          <-- controller-gen reads markers, writes DeepCopy methods
         |
         v
    make manifests         <-- controller-gen reads markers, writes CRD/RBAC/webhook YAML
         |
         v
    make docker-build      <-- Go compiler builds your code + controller-runtime into a binary
         |
         v
    make install           <-- kubectl applies the CRDs controller-gen generated
         |
         v
    make deploy            <-- kustomize composes everything, kubectl applies it
         |
         v
    Operator pod starts    <-- controller-runtime's Manager boots, controllers reconcile
```

Kubebuilder generated the Makefile that orchestrates this pipeline. Controller-gen generates the artifacts. Controller-runtime runs the operator. Each tool has a clear boundary.

If you add a new field to your types, the pipeline is: `make generate` (controller-gen updates DeepCopy), `make manifests` (controller-gen updates CRD YAML), `make install` (kubectl applies new CRD), restart operator. Missing a step produces confusing errors: your CRD rejects valid objects (forgot `make manifests`), your operator panics on serialization (forgot `make generate`), your operator gets "forbidden" from the API server (forgot to deploy updated RBAC from `make manifests`).

## When Things Go Wrong

Each failure maps to a specific tool.

**Kubebuilder failures:**

- *You delete a `// +kubebuilder:scaffold:` comment from main.go.* These are kubebuilder's insertion points. The next `create api` command fails. Fix: add the marker back, or manually add the import, scheme registration, and controller setup.
- *You scaffold a webhook but admission never fires.* Kubebuilder generated the code, but `config/default/kustomization.yaml` still has the webhook and cert-manager sections commented out. The manifests exist but aren't included in `make deploy`. Fix: uncomment the relevant sections.

**Controller-gen failures:**

- *You forget `make generate` after adding a type.* Your code compiles until something uses it through a `runtime.Object` interface. Cryptic error: "does not implement runtime.Object." Fix: `make generate`.
- *You forget `make manifests` after adding fields.* Your Go types have new fields but the CRD in the cluster doesn't know about them. The API server silently drops unknown fields. Your operator reads the object back and the fields are gone. Fix: `make manifests && make install`.
- *Your RBAC markers are stale.* You start creating Deployments but forgot to add the RBAC marker. Your operator gets `forbidden: User "system:serviceaccount:..." cannot create resource "deployments"`. Fix: add the marker, `make manifests`, redeploy.
- *CRD validation rejects valid objects after an upgrade.* You tightened a marker (lowered a Maximum, added Required) but existing objects violate the new constraint. Fix: migrate existing objects first, or use a webhook for validation instead (webhooks only run on create/update, not on stored objects).

**Controller-runtime failures:**

- *Your Reconcile panics.* Controller-runtime catches it. The panic becomes an error. The item requeues with backoff. Other workers keep running. The process doesn't crash.
- *The leader loses its lease.* The manager exits immediately. The new leader re-lists everything and processes it. Level-triggered reconciliation means nothing is lost -- the system converges.
- *The cache serves stale data.* You update an object, immediately read it back, and get the old version. Then the watch event arrives, the object is re-enqueued, and Reconcile runs again with fresh data. Convergence, not correctness-on-first-try.

## The Unifying Idea

Kubebuilder is scaffolding that writes itself out of your project. Controller-gen is the generator that reads your markers and produces manifests. Controller-runtime is the framework that runs in your cluster. Three tools, clear boundaries, one operator.

Next time you run `kubebuilder create api`, you'll know exactly what happens. Kubebuilder writes Go files and modifies main.go. Controller-gen reads the markers and generates YAML. Controller-runtime runs the Reconcile function you fill in. The scaffolding disappears. The generator runs at build time. The framework takes over at runtime. Your operator converges.
